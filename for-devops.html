L2 — Developer DX

@ComplianceRequired — guard omission is a build failure

The Cursor/MCP IDE agent loads generated guards at coding time. Annotate any method with @ComplianceRequired and the agent enforces correct guard usage — both as a lint warning and a compiler-enforced build failure.

L0 + L1 — MDD Pipeline

Verified model → generated guards. Zero hand-written compliance code.

The MDD engine reads the OCL-verified threat model and auto-generates Java guards, Solidity on-chain registry, and MCP stubs. Non-compliant models cannot produce deployable artifacts. The threat model IS the source of truth.

Generated artifacts from verified .ecore model:
✓ MITREController.java — 27 threat guards
✓ GDPRController.java — consent + purpose guards
✓ AuditRegistry.java — on-chain record() calls
✓ AuditRegistry.sol — Solidity on-chain contract
✓ compliance-mcp-stubs/ — IDE agent integration
OCL invariant · threat-model.ecore
-- OCL invariant: every HIGH-risk threat -- must have at least one active mitigation context ThreatScenario inv GDPRCoverage: self.riskLevel = RiskLevel::HIGH implies self.mitigations->exists(m | m.status = MitigationStatus::ACTIVE ) -- All valid consent claims must reference -- an existing data subject consent record context ConsentClaim inv ValidConsents: self.dataSubjects->forAll(ds | ds.consents->notEmpty() and ds.consents->exists(c | c.isValid()) ) -- Violation → model rejected → no codegen
